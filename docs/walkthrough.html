Curb

One local service that watches how much your AI coding agents are spending — tokens, models, sessions — then warns and, only when you ask it to, stops a runaway worker. The CLI and the web dashboard are thin clients of that service. No prompt or response content is ever captured.

A single Rust binary. cargo build --release --bin curb → curb app serves an embedded React dashboard on loopback and opens your browser. Rust owns everything stateful; the UI just renders.

01The big idea

An agent process can sit idle for hours burning nothing. A tight agent loop can vaporize a model budget in minutes. So Curb measures spend, not clock time.

Tokens are the risk primitive

Wall-clock runtime is demoted to a stale/stuck-process hint. The first question Curb answers is “what is spending tokens right now, what is idle, what is risky?”

Metadata only, ever

Curb reads the local usage logs Codex & Claude already write — token counts, model ids, session ids, cwd, timestamps. Prompt/response text, files, screenshots, and keystrokes are out of scope and rejected at config load.

The endpoint owns policy

No cloud control plane. One local service correlates usage to live processes, evaluates policy, notifies, and (only in enforcement mode) terminates — then writes an append-only audit ledger.

The one metric: turn spend

Turn spendTokens an agent has consumed since your last input. Codex user_message events and Claude typed-input rows mark turn boundaries; Curb sums the fresh spend after the latest boundary. This single number drives the headline and the warn/kill lines.
Spent, not contextSpend = uncached input + cache-creation + output + reasoning. Cached reads are excluded — re-reading a big context isn't fresh work. A finished expensive turn that's now idle does not alert.

Every agent is described by three facts: turn_tokens (spend this turn), status (working or idle), and alert (ok, warn, or kill). That's the whole vocabulary — there is no checkpoint/window/recency jargon.

Two operating modes

ModeWhat it doesCan it kill?

02Architecture at a glance

A deep daemon surrounded by thin clients. The clients sort, filter, select, and render — they never parse logs, inspect process trees, recompute policy, or decide what's safe to kill.

The deep-module betEach src/*.rs file hides a hard problem behind a small surface: provider log quirks, process-identity safety, policy thresholds, the audit ledger. Clients compose those modules instead of reimplementing their internals. Information-hiding is the architecture — Tauri/tray/installers are just future packaging around the same local API.

Incremental ingestion (the part that makes it cheap to leave open)

The usage::Reader caches parsed events per file. On each scan it compares (size, mtime); unchanged files return cached events instantly. When a file grew, it verifies the old prefix still hashes the same (catching truncation/replacement) and parses only the appended bytes. The cache is persisted to usage-cache.json with 0600 perms, so a daemon restart resumes mid-stream instead of re-reading everything.

size+mtime fast path prefix-hash tamper check append-only re-parse live-tail last 256KB deleted-file pruning per-provider error isolation

Provider adapters are allowed to be uneven. Codex and Claude expose rich local metadata today; a bad row in one provider becomes a source-health error, never a fatal dashboard failure — the other provider still loads.

05The state model: two words plus a flag

Each agent carries two independent state words and one capability flag — because “is it spending now” and “what can Curb do about it” are genuinely different facts. This replaced an eight-value, three-axis vocabulary that nobody could read at a glance.

status activity

Is it spending right now?

  • working fresh checkpoint just landed
  • idle alive/recent, not spending now

alert vs your lines

Where is this turn vs warn/kill?

  • ok within limits (or acknowledged)
  • warn this turn crossed warn
  • kill this turn crossed kill

can_stop safety gate

Could Curb safely stop it now?

  • true enforce mode · correlated · enforceable · over kill
  • false anything else — the row explains why
A blocked kill is a success, not an errorWhen alert is kill but can_stop is false — uncorrelated, a watch-only desktop app, or warn-only mode — the row says so in one plain sentence. Refusing to kill the wrong thing is the feature.

Correlation: matching spend to a living process

service.rs scores every session against every matched process. The signal that links them is provider + working directory:

ScoreEvidenceReason tag
125provider matches & cwd is an exact matchprovider+cwd
75provider matches & cwd is a safe prefix (≥2 path components each side; rejects / & /Users)provider+cwd-prefix
0provider mismatch or no usable cwd → uncorrelated—

Process↔agent confidence is additive and separate: process_name +50, command_regex +40, executable_path +80, bundle_id +80; exclusion filters reset a match to zero. The matched evidence types are surfaced verbatim to the UI in matched_by so an operator can see why Curb thinks pid 1234 is this session.

Ranking: spend outranks runtime, always

Sessions sort by risk_rank (0 = stop-pending, 1 = warn/would-stop/blocked, 2 = acknowledged, 3 = idle-high, 5 = quiet) → then latest-turn tokens → window tokens → recency. A six-hour-old idle process never jumps above a session that just spent a million tokens.

06Policy & the watch loop

A background thread ticks on the usage scan interval. Each tick rebuilds sessions, correlates, and runs the usagewatch policy engine. State lives in three little maps: warned, grace, and targets.

The rolling window matters

Policy evaluates recent activity only. A session with a giant historical turn but zero tokens inside the current window (default 15m) shows up in the dashboard as quiet-high / idle-high — visible, but never warned or enforced. Runaway means spending now.

Presets: one command to a sane posture

Config is strict: #[serde(deny_unknown_fields)], warn < stop enforced, every matcher regex must compile at load, and ledger.include_prompt_content: true is a hard validation error. Writes are atomic (temp file + rename) with 0600/0700 permissions.

07The safety gauntlet

Termination is the one irreversible thing Curb does, so it's wrapped in layers. A kill only happens when every gate passes — and the platform layer refuses to accept a bare PID at all.

Mode is enforcement
Visibility & alert physically cannot reach the terminate call — they emit would-stop.
Latest turn crossed the stop threshold
Not historical, not cumulative — the most recent checkpoint, inside the window.
No active acknowledgement
A user ack (bounded, capped at the configured extension) suppresses the crossing entirely.
Grace period elapsed
A timer starts on first crossing; the session gets a window to quiet down or be acknowledged.
Correlated to a live, enforceable worker
Uncorrelated → blocked. A watch-only desktop app root is never a target — only worker/CLI processes.
Identity revalidated at the instant of the kill
PID + start time + owner + executable/bundle/team must still match. PID reuse → refuse.
Sealed TerminationTargetThe platform API cannot be handed a raw PID. The only way to get a TerminationTarget is Snapshot::termination_target(process), which fails unless the process has pid ≥ 2, a real start time, an owner, and at least one of executable/bundle/team id. The target carries the whole process subtree in children-first order. Termination is SIGTERM to leaves first → wait grace → SIGKILL to survivors (taskkill /T /F on Windows).

Hardening

loopback-only (double-checked at bind) 64-char token, 0600 file constant-time compare same-origin CSRF guard on cookie writes CORS limited to loopback origins 64KB headers · 1MB body caps
Why curb app needs no token pasteThree auth paths exist: Authorization: Bearer, an X-Curb-Token header, and a curb_token cookie (HttpOnly · SameSite=Strict · Path=/v1/). The embedded dashboard is served same-origin and gets the cookie automatically, so the browser just works. Bearer/header auth is for separate API clients. Unsafe cookie-authed requests additionally require a matching Origin.

Renders from selectors

Sorting/grouping lives in tested read-model functions (selectOperatorSummary, sessionActivityRows), not ad-hoc component branches. The UI re-asks the daemon — it doesn't recompute policy.

Demo-safe by default

Before connecting, it shows clearly-marked demo data so it can never be mistaken for live protection. It polls /v1/snapshot every 5s and degrades to a readable error on connection loss.

Progressive disclosure

Spend & risk are the first viewport. Connection plumbing, onboarding, drill-downs, and policy settings live in collapsible drawers below the fold.

build.rs walks web/dist and emits an include_bytes! table, so the binary serves the whole SPA with zero filesystem reads at runtime. cd ui && npm run dev is only needed while developing the frontend.

11The audit ledger

Every meaningful event — service start, policy warning, ack, grace, termination success/failure — is appended to a tamper-evident NDJSON journal. It's evidence, scrubbed of anything sensitive.

// one line per event, hash-chained, content-scrubbed
{ "type":"usage_warning", "seq":42, "ts":"2026-05-29T22:00:00Z",
  "agent_id":"codex-cli", "mode":"alert",
  "message":"latest turn 1.4M > warn 1.0M",
  "prev_hash":"9c1f…", "event_hash":"a73b…" }
  • Monotonic seq + prev_hash→event_hash SHA-256 chain detects tampering or gaps.
  • scrub_sensitive_data recursively strips anything that smells like content before write.
  • Owner-only file perms; launch config keeps the audit ledger local-only.

Event types include:

service_startedrun_started policy_warningusage_warning ack_receivedsession_ack_received usage_termination_started usage_termination_completed usage_termination_failed usage_scan_failed

12Build, verify, ship

Rust edition 2024, a lean dependency set, and a single pre-merge gate that builds the UI, lints, type-checks, and runs every test on both sides.

From source

cargo build --release --bin curb
./target/release/curb install
curb app                       # serve embedded UI + open browser

The gate — scripts/validate.sh

build-ui.sh --check          # UI assets are fresh & embedded
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo test                   # 140 Rust tests
run-backlog-006-demo.sh --dry-run   # synthetic end-to-end demo
cd ui && npm run typecheck && npm run lint && npm test  # 39 UI tests

Dependencies kept thin

anyhow · thiserror · clap · serde(+json/yaml) · chrono · regex · url · sha2 · hex · getrandom · sysinfo. No async runtime — the HTTP server is a deliberately small blocking loop.

Tested with real things

Real temp config files, real temp ledgers, harmless real subprocesses, and deterministic process-snapshot fixtures. Mocks only at the true OS boundary (notifications, termination).

The standing invariantsVisibility & alert never terminate · prompt/response/file capture is rejected by default · PID + start time is the identity boundary · desktop app roots are never enforcement targets · termination APIs never accept a bare PID. These aren't guidelines — they're enforced by the type system and the test suite.
