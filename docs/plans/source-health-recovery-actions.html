Roadmap item 1 / source-health recovery

Make provider source-health failures actionable without making them readiness failures.

The refreshed long sidecar removed the periodic readiness blocker but still recorded repeated sanitized Codex source-health errors. This slice keeps readiness green, preserves privacy, and upgrades the recovery item from a generic command into a source-specific operator next step.

Chosen Design

Keep the existing service-owned recovery surface. Deepen curb-core/src/service/recovery.rs so sanitized source errors classify into concrete status, action, command, and runbook guidance.

Acceptance

Given provider source reports with invalid UTF-8, JSON, oversized-line, symlink, outside-root, or permission errors, the snapshot exposes sanitized provider recovery with no raw local paths or payload words. Overview status remains driven by session alerts, not source-health errors.

Verify

Red then green Rust service tests, focused UI recovery test if wire text changes, the long-sidecar verifier against the June 15 packet, and scripts/validate.sh.

Stop If

Provider recovery needs raw file paths, raw payloads, or provider-specific filesystem mutation. Curb must advise and report; it must not repair provider logs automatically.
